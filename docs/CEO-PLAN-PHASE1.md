## Vision

### 10x Check
The Phase 1 scope (4 milestones covering DICOM import, patient management, RT visualization, contouring) is the right foundation. The "10x" for this project is not a bigger scope — it's **architectural correctness at the foundation** so Phase 2+ can build on it without re-architecting. Key 10x investments:
- Cornerstone3D Segmentation Service instead of Canvas2D (enables DICOM SEG export, 3D, AI labelmap in Phase 2)
- JWT auth from day 1 (avoids retrofitting auth to a system already handling PHI)
- Full AI HTTP interface, not a stub (Phase 2 AI integration is a drop-in, not a new feature)

### Platonic Ideal
A radiation oncologist opens the TPS, creates a patient, uploads a DICOM folder, and within 2 minutes is looking at their CT with RT Structure contours overlaid and RT Dose heatmap visible. They can toggle structures on/off, adjust window/level, and brush-edit a contour. The AI auto-contour button shows "not configured" but the UI for it is complete. The system is fast, the UI follows the dark clinical aesthetic from DESIGN.md, and everything feels inevitable.

## Key Architectural Decisions

1. **Cornerstone3D Segmentation Service** — NOT Canvas2D overlay for contouring. This is the most important architectural decision in the plan. It changes Milestone 4 entirely.

2. **Client-side DICOM rendering** — `wadouri.fileManager` + Blob URLs. NOT a WADO server. Correct call for Phase 1 complexity level.

3. **AI Auto-Contouring as HTTP Hook** — `POST /api/contouring/auto` is real. The Phase 1 "not configured" response is intentional — it's a 501, not a console.log.

4. **Auth from Day 1** — Even though it's Phase 1 prototype, PHI exposure risk justifies the ~1 day investment.

5. **Phase 2 single-origin deployment** — Documented now so `uploads/` path and CORS decisions don't need to change in Phase 2.
